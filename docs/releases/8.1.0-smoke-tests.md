## 0. How to record results

1. Pick a single commit of `siropkin/budi` `main` at or after the 8.1 merge point as the tag candidate.
2. Build `cargo build --release --locked` on the target platform. The smoke run must use the same binary the release artifacts will ship.
3. For each test you run, capture: test ID, OS, pass/fail, a short evidence blob (trimmed command output, screenshot, or log snippet).
4. Post the results as a single comment on **[#297](https://github.com/siropkin/budi/issues/297)** using the [results template](#14-results-reporting-template).
5. Any FAIL that is not a genuine blocker (flake, missing prereq, unrelated env issue) must be re-run before close; only a **full PASS on the minimum-viable set** clears the gate.

## 4. R1.4 â€” File-level attribution (#292)

### ST-81-FL-01: File paths are repo-relative and inside the repo

**Goal.** Every file recorded by the R1.4 enricher is repo-relative, never absolute, never outside the repo, never file content.

**Procedure.**

```bash
budi launch claude    # (any agent that emits tool calls will do)
# Have the agent touch a file inside the current repo, e.g. "read src/main.rs"
budi stats --files -p today
budi stats --file src/main.rs
curl -s http://127.0.0.1:7878/analytics/files | jq .
# Pull the raw rows to inspect path format:
sqlite3 ~/.local/share/budi/analytics.db \
  "SELECT path FROM files LIMIT 50;"
```

**Pass criteria.**
- [ ] `budi stats --files` lists recently-touched files ranked by cost / tool-activity.
- [ ] `budi stats --file <path>` returns per-ticket / per-branch / per-activity / per-tool breakdown for that file.
- [ ] **No row** in `files.path` starts with `/`, `~/`, or a drive letter (`C:\`, etc.).
- [ ] **No row** contains `..` segments pointing outside the repo.
- [ ] **No row** contains file contents, diff snippets, prompts, responses, or any free-form text that could carry secrets.
- [ ] File identity joins cleanly with ticket / branch / activity / tool dimensions in the analytics API.
- [ ] When the agent references a path that does not resolve cleanly (symlink to outside the repo, non-existent file), the row is dropped rather than stored as a placeholder.

### ST-81-FL-02: Filesystem paths from mixed tool calls normalize deterministically

**Goal.** Repeated tool calls against the same file from different working directories collapse to one canonical repo-relative path.

**Procedure.**

```bash

budi stats --file src/main.rs -p today
```

**Pass criteria.**
- [ ] All three references collapse to a single row keyed by `src/main.rs` with the aggregated cost / tool counts.
- [ ] No duplicate rows with different path spellings.

## 5. R1.5 â€” Tool outcome and sessionâ†’work outcome (#293)

### ST-81-OC-01: Tool-call outcomes are bounded and debuggable

**Goal.** Every tool call is stamped with one of the fixed outcomes and an explicit `source` / `confidence`.

**Procedure.**

```bash
# Drive a session that exercises success, error, denied, and retry paths:
#   1. A successful read (success)
#   2. A bash command that exits non-zero (error)
#   3. A tool call the agent refuses / the user denies (denied)
#   4. A transient failure followed by a retry in the same session (retry)
budi stats --tools -p today
sqlite3 ~/.local/share/budi/analytics.db \
  "SELECT name, outcome, source, confidence FROM tool_outcomes ORDER BY ts DESC LIMIT 50;"
```

**Pass criteria.**
- [ ] Every row's `outcome` is one of `success`, `error`, `denied`, `retry`. No free-form strings.
- [ ] Every row has a non-null `source` (e.g. `tool_result`, `assistant_text_heuristic`, etc.) and a `confidence` in `[0.0, 1.0]`.
- [ ] No row contains tool inputs, tool results, or error message bodies â€” only enumerated outcomes, identifiers, and counts.
- [ ] `budi sessions <id>` renders a tool-outcome row on the session detail view that matches the DB counts.

### ST-81-OC-02: Session â†’ work outcome derived locally from git

**Goal.** Every session gets a `work_outcome` in `committed` / `no_commit` / `branch_merged` / `unknown`, computed from **local git state only**.

**Procedure.** In a repo with at least one session that produced a commit, one session with no commit, and one branch that was merged into the integration branch since the session ended:

```bash
budi sessions -p week --format json | jq -r '.[] | [.id, .work_outcome] | @tsv'
# Spot-check that no daemon subprocess made outbound HTTP to GitHub / GitLab / Bitbucket during classification:
grep -Ei 'github\.com|gitlab\.com|bitbucket\.org' ~/.local/share/budi/logs/daemon.log || echo "no remote git/PR API calls (OK)"
```

**Pass criteria.**
- [ ] Every session in the window has exactly one of the four work outcomes.
- [ ] Outcomes match local git reality: sessions with post-session commits on the active branch â†’ `committed`; sessions whose branch later merges into the integration branch â†’ `branch_merged`; etc.
- [ ] Daemon log for the window shows **no** outbound HTTP calls to `github.com`, `gitlab.com`, `bitbucket.org`, or any remote git / PR API host.
- [ ] `source` / `confidence` are present on the work-outcome row in the session detail view.

### ST-81-CL-04: `budi cloud sync` â€” disabled / empty / conflict / JSON


**Pass criteria.**
- [ ] (a) Exit 2; body includes `result: "disabled"`; no network call.
- [ ] (b) Exit 0; body includes `result: "empty_payload"`; human text reads `Nothing new to sync`.

- [ ] (d) JSON includes `ok`, `result`, `endpoint`, `records_upserted`, `rollups_attempted`, `sessions_attempted`, `watermark`; exit code matches `ok`.

### ST-81-CL-05: Daemon loopback gate and public status

**Pass criteria.**
- [ ] `POST /cloud/sync` from a non-loopback source returns `403 Forbidden`, matching the `/sync` admission contract.

## 8. R2.3 â€” Simplified statusline (#224)

### ST-81-SL-01: Default statusline is quiet and provider-scoped

**Goal.** The default `budi statusline` output is intentionally simple, reflects **rolling** `1d` / `7d` / `30d` windows (not calendar), and provider-scopes to the caller's provider â€” no blended totals.

**Procedure.** On a machine that has both Claude Code and Cursor usage today:

```bash
budi statusline
budi statusline --format claude            # auto-scoped to claude_code
budi statusline --format json | jq .
budi statusline --format json --provider cursor | jq .
```

**Pass criteria.**
- [ ] Default slot list renders `1d` / `7d` / `30d`, in that order, in the human-readable output.
- [ ] `budi statusline --format claude` never returns Cursor / Codex / Copilot spend â€” it is strictly scoped to `claude_code`.
- [ ] `budi statusline --format json` emits all three of `cost_1d` / `cost_7d` / `cost_30d`, the `provider_scope` field echoes the filter, and deprecated aliases `today_cost` / `week_cost` / `month_cost` are populated with the same rolling values.
- [ ] `--provider cursor` scopes all numeric fields (`cost_1d` / `cost_7d` / `cost_30d`, `session_cost`, `branch_cost`, `project_cost`) to Cursor only.
- [ ] `active_provider` reflects the most recent provider seen in the 1d window **after** the filter is applied.
