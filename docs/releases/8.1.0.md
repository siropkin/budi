## 1. Release theme

Budi 8.1 is the release that makes **local Budi something enterprise developers love using every day**. It is not a breadth release. 8.0 laid the proxy-first foundation; 8.1 turns that foundation into daily-use polish:

- **Trustworthy attribution first** — sessions, branches, tickets, and activities are fixed and made queryable before any richer signals are added (R1.0).
- **Richer local signals inside privacy limits** — file-level attribution and tool / session outcome signals stay repo-relative, rule-based, and debuggable (R1.4, R1.5).
- **A simpler daily-use surface** — one quiet provider-scoped statusline centered on `1d` / `7d` / `30d`, a normalized CLI, and a first-run path that is trustworthy without a cheat sheet (R2).
- **Local and cloud telling the same story** — the cloud dashboard adopts the same windows and owns the local→cloud linking flow; the Cursor extension consumes the shared provider-scoped status contract (R3).

See [ADR-0088](../adr/0088-8x-local-developer-first-product-contract.md) for the governing product contract.

## 5. Release notes draft (for the GitHub release body)

> Paste the body below into the `v8.1.0` release when R4.3 runs. Final commit hash, date, and cross-repo version numbers are filled in at tag time.

### Budi 8.1 — local Budi you actually want to use every day

Budi 8.1 is focused on one thing: making local Budi feel polished enough that enterprise developers leave it running on every machine they code on. It does not add new agent coverage and it does not add manager-facing budgets — both are called out explicitly under "Deferred" below.

The theme is **developer-first local polish**: trustworthy attribution first, richer local signals inside the ADR-0083 privacy limits, a simpler quieter daily surface, and cloud and Cursor surfaces that tell the same story as the CLI instead of forking it.

Governed by [ADR-0088](https://github.com/siropkin/budi/blob/main/docs/adr/0088-8x-local-developer-first-product-contract.md).

#### Install

**macOS / Linux (Homebrew)**

```bash
brew update && brew upgrade siropkin/budi/budi
budi doctor
```

**macOS / Linux (install script)**

```bash
curl -fsSL https://raw.githubusercontent.com/siropkin/budi/main/scripts/install.sh | sh
```

**Windows**

Download `budi-v8.1.0-x86_64-pc-windows-msvc.zip` from the assets and add the extracted folder to `PATH`. All assets are checksummed in `SHA256SUMS`.

After upgrading, `budi doctor` is the single source of truth for "did the upgrade land cleanly?".

#### Attribution reliability first (R1.0)

Four real bugs surfaced against a live user's local Budi were the hard gate for 8.1; richer signals don't land on top of silent attribution failures.

- **`budi sessions` now reflects today's proxy activity (#302).** `insert_proxy_message` dropped the `session_id` column, so every proxied assistant message was written with `session_id = NULL` and filtered out. Both the live proxy path and the defensive analytics filter now treat empty-string `session_id` as `NULL`. A new `budi doctor` sessions-visibility check flags any window where activity exists but zero sessions are returned. A new shell-driven end-to-end regression (`scripts/e2e/test_302_sessions_visibility.sh`) fails when the fix is reverted.
- **Live branch attribution works end-to-end again (#303).** `budi stats --branches` no longer collapses everything into `(untagged)`; proxy-observed branches flow into the pipeline and analytics join without being rewritten.
- **Ticket is a first-class CLI dimension (#304).** `budi stats --tickets` and `budi stats --ticket <id>` now exist and match how enterprise developers slice work. `/analytics/tickets` exposes the same shape.
- **Activity is a first-class CLI dimension (#305).** `budi stats --activities` and `budi stats --activity <name>` match the new CLI shape for tickets. `/analytics/activities` exposes the same shape.

The attribution contract is documented in `SOUL.md` so future providers cannot silently regress R1.0.

#### Richer local signals inside the privacy contract (R1.4 / R1.5)

- **File-level attribution (#292).** Tool-call file paths are extracted, normalized to **repo-relative paths only**, and joined with ticket / branch / activity. No absolute paths, no outside-of-repo paths, no file contents, no diffs. A path that doesn't resolve cleanly is dropped rather than stored as a placeholder. New CLI: `budi stats --files`, `budi stats --file <path>`. New API: `/analytics/files`. Fully inside ADR-0083.
- **Tool and work outcomes (#293).** Every tool call records a bounded outcome (`success` / `error` / `denied` / `retry`). Every session gets a work outcome (`committed` / `no_commit` / `branch_merged` / `unknown`) derived from local git state only — no remote git or PR API calls. Every outcome signal carries an explicit `source` / `confidence` so surfaces can say "this is a heuristic" honestly. Surfaced on the session detail view.

#### Classification contract (R1.1–R1.3)

- **ADR-0088** locks the 8.x local-developer-first product contract (#216).
- **Activity classification (#222)** gets explicit `source` and `confidence` signals, wider provider and ingest coverage, and shared pipeline logic. No new local LLM dependencies on the primary 8.1 classification path.
- **Ticket attribution and branch classification (#221)** become first-class analytics (not best-effort side effects) with the same `source` / `confidence` surfacing.

#### A simpler quieter daily surface (R2)

- **`budi cloud sync` and `budi cloud status` (#225).** The missing self-serve cloud sync command is back under a discoverable `budi cloud` namespace, in line with the normalized command / flag set. Both commands honor `--format text|json` like the rest of the CLI.
- **First-run UX reshaped (#228).** The post-`budi init` "Next steps" list now follows the order users actually need: restart-terminal → start coding → `budi doctor` as the single end-to-end verifier → `budi status` → optional `budi import`. `budi status` and `budi doctor` now print friendly first-run nudges instead of rendering all-zero summaries when the daemon is healthy but no activity is recorded yet. Install scripts and the README all now point at `budi doctor` as the canonical "did setup really work?" command. Cross-surface local→cloud linking UX is deliberately **not** in this ticket — it lives in the dashboard flow under R3.
- **Quiet provider-scoped statusline on `1d` / `7d` / `30d` (#224).** The default statusline is intentionally simple. Rolling `1d` / `7d` / `30d` replaces calendar today / week / month on this surface (`budi stats` still uses calendar semantics). `budi statusline --format claude` is auto-scoped to `claude_code` — it no longer blends Cursor / Codex spend into the Claude Code status line (the 8.0 blended-totals bug this ticket was opened against). A single shared provider-scoped status contract is now defined in [`docs/statusline-contract.md`](https://github.com/siropkin/budi/blob/main/docs/statusline-contract.md) and consumed by the CLI, the Cursor extension, and the cloud dashboard. Deprecated aliases `today_cost` / `week_cost` / `month_cost` are populated with the same rolling values for one release of backward compatibility and will be removed in 9.0.
- **Cursor extension as onboarding entry point (#314).** Marketplace-first installs flow through a welcome view in the extension that hands off to `budi init`. `budi doctor` now surfaces a one-line summary of local-only onboarding counters (welcome-view impressions, "Open Terminal" clicks, completed hand-offs). Zero remote telemetry — the counters file stays local.

#### Surface alignment (R3)

- **Cloud dashboard windows + local→cloud linking (#235).** The dashboard adopts the same `1d` / `7d` / `30d` windows as local and owns the local→cloud linking flow end-to-end: discoverable linking after signup, automatic initial sync on successful link, sync freshness indicator, and visibly distinct `not_linked` / `linked_no_data` / `ok` / `stalled` states so a freshly linked account is never indistinguishable from a broken one.
- **Cursor extension alignment (#232).** The extension is now a provider-scoped statusline-only surface that mirrors the Claude Code statusline byte-for-byte with `?provider=cursor`. It consumes the shared status contract from #224 rather than inventing a separate shape.

#### Scope explicitly **not** shipped in 8.1

These are tracked and in motion, just not in this release. Please file under the linked tickets rather than against 8.1.

- **Broader AI coding tool coverage** (Gemini CLI, Windsurf, Cline, Aider, Roo Code, Continue) → tracked under #294 (now targeted at 8.3 after the 8.2 pivot to JSONL tailing — see ADR-0089 and #316).
- **Invisible-Budi JSONL tailing** replacing the proxy-first live path → 8.2 (#316). 8.2 does not start until 8.1 is tagged and promoted.
- **MCP server reintroduction** → 9.0 (#162 under #159).
- **Budgets, limits, pipeline integrations, richer ticket metadata, richer team workflows** → 9.0 (#159).
- **Uploading prompts, code, or responses to the cloud** — intentionally out of scope in every 8.x release. ADR-0083 is the privacy contract.

#### Upgrade notes

- `budi upgrade` / `brew upgrade siropkin/budi/budi` is the canonical path. No manual schema edits are required; run `budi doctor` after upgrading.
- Custom `~/.config/budi/statusline.toml` files using the pre-8.1 slot names (`today` / `week` / `month`) keep rendering — the CLI normalizes them to the new canonical `1d` / `7d` / `30d` slots at load time.
- `today_cost` / `week_cost` / `month_cost` fields in `budi statusline --format json` are still populated (with the rolling values) for one release of backward compatibility and will be removed in 9.0.
- The Cursor extension and cloud dashboard are shipped from their own repos (ADR-0086); upgrade those surfaces via their normal distribution channels.

#### Repo coordination

- Daemon: `siropkin/budi` — v8.1.0 (this release).
- Cursor extension: `siropkin/budi-cursor` — v1.2.x (statusline-only + onboarding counters).
- Cloud dashboard: `siropkin/budi-cloud` — dashboard aligned with the `1d` / `7d` / `30d` window contract and owning local→cloud linking.
- Public marketing site: `siropkin/getbudi.dev` — synced per #296 before this tag is promoted.

#### Thanks

8.1 is a tight developer-polish release; thank you to everyone who ran local Budi against real work and reported the attribution bugs that became R1.0.
